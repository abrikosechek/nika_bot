# Hands Bot

Discord-бот на `discord.py` с модульной архитектурой (`cogs`), конфигурацией на уровне серверов и runtime-логикой для приватных голосовых комнат.

## Назначение проекта

Проект ориентирован на быстрый рост функционала без усложнения ядра:

- новые фичи добавляются как отдельные модули (`cogs/*.py`);
- поведение привязывается к конфигам, а не к хардкоду;
- критичные voice-сценарии (`private`, `follow`) изолированы и предсказуемы.

## Технологический стек

- Python 3.11+
- `discord.py` 2.x (slash-команды + events)
- `python-dotenv`
- JSON-конфиги (`config/guilds.json`, `config/global.json`)

## Быстрый старт

1. Активировать виртуальное окружение.
2. Убедиться, что в `.env` есть токен (`SECRET`).
3. Запустить бота:

```powershell
py .\main.py
```

При успешном запуске бот:

- подключается к Discord Gateway;
- загружает все cogs из директории `cogs`;
- синхронизирует slash-команды.

## Структура проекта

- `main.py` — точка входа, инициализация бота и загрузка модулей.
- `cogs/` — функциональные модули:
  - `fun.py` — игровые slash-команды (`/dice`, `/bananza`);
  - `private.py` — приватные голосовые комнаты и команды управления;
  - `follow.py` — voice-follow для `god_user_id`.
- `utils/config.py` — загрузка и доступ к конфигам.
- `utils/decorators.py` — переиспользуемые проверки/guard-логика.
- `config/guilds.json` — настройки по серверам.
- `config/global.json` — глобальные настройки бота.

## Архитектурные принципы

### 1) Modular by default

Каждый модуль решает ограниченную задачу и подключается через `setup(bot)`.

### 2) Config-driven behavior

ID каналов и режимы задаются в JSON-конфигах, а не в коде.

### 3) Slash-first интерфейс

Пользовательское взаимодействие строится вокруг slash-команд, с валидацией контекста и понятными ответами.

### 4) Defensive voice handling

Для voice-логики критично избегать гонок и конфликтов между модулями (`follow` и `private`), особенно в событиях `on_voice_state_update`.

## Конфигурация

### `config/guilds.json` (per-guild)

Пример:

```json
{
  "1303975959472635944": {
    "fun": {
      "text_channel": 1303975959472635947
    },
    "private": {
      "control_text_channel": 1481821296181383404,
      "create_voice_channel": 1481821335997911073,
      "rooms_category": 1481821400000000000
    },
    "verification": {
      "message_id": 1481821500000000000,
      "verified_role_id": 1481821600000000000,
      "unverified_role_id": 1481821700000000000
    }
  }
}
```

Поля `private`:

- `control_text_channel` — канал, где разрешены команды `/private ...`;
- `create_voice_channel` — вход в этот voice-канал триггерит создание личной комнаты;
- `rooms_category` — категория, где создаются приватные комнаты.

Поля `verification`:

- `message_id` — ID сообщения для верификации;
- `verified_role_id` — роль, выдаваемая после реакции на сообщение верификации;
- `unverified_role_id` — роль, выдаваемая пользователю при входе на сервер.

Поведение fallback:

- если секция `verification` отсутствует, модуль верификации на этом сервере ничего не делает;
- если отсутствует часть полей или роль не найдена, выполняется доступная часть логики без ошибок.

### `config/global.json` (global)

Пример:

```json
{
  "god_user_id": 698918730922721281
}
```

- `god_user_id` используется модулем `follow` как целевой пользователь для следования.

## Модуль `private`: как работает

### Runtime state

Состояние хранится в памяти:

- `rooms_by_channel_id` — карта `room_channel_id -> RoomState`;
- `owner_to_room_channel_id` — карта `owner_id -> room_channel_id`.

`RoomState` включает:

- `owner_id`
- `name`
- `is_locked`
- `user_limit`
- `banned_user_ids`
- `owner_left_at`

### Lifecycle

1. Пользователь заходит в `create_voice_channel`.
2. Если у него уже есть комната — он перемещается в нее.
3. Если комнаты нет — создается новая, затем пользователь перемещается в нее.
4. При выходе owner фиксируется `owner_left_at` (если в комнате кто-то остался).
5. При возврате owner `owner_left_at` сбрасывается.
6. Если в комнате не осталось людей, комната удаляется и state очищается.

### Команды `/private`

- `lock` / `unlock`
- `limit`
- `kick`
- `ban` / `unban`
- `close`
- `claim`
- `info`

Особенности:

- команды доступны только в `private.control_text_channel`;
- ответы отправляются как `ephemeral`;
- сообщения об ошибках возвращаются в формате `Ошибка: ...`.

## Модуль `follow`: важное ограничение

`follow` отключается для private-контекста:

- не реагирует на `private.create_voice_channel`;
- не реагирует на активные private-room каналы.

Это предотвращает конфликты connect/move между `follow` и `private`.

## Типичный поток обработки событий voice

Событие `on_voice_state_update` маршрутизируется через `private`-модуль, где:

1. сначала обрабатывается интеграция с `follow`;
2. затем применяется lifecycle приватных комнат;
3. выполняются cleanup-правила (включая удаление пустых комнат).

## Добавление нового модуля (чеклист)

1. Создать `cogs/<module>.py`.
2. Реализовать класс `commands.Cog` и `async def setup(bot)`.
3. Добавить slash-команды и/или event-listeners.
4. Если нужны настройки — добавить секцию в `guilds.json` или `global.json`.
5. Добавить guard-проверки (guild-only, канал, валидность входных данных).
6. Проверить, что модуль не конфликтует с существующими voice-сценариями.

## Поддержка и развитие

Рекомендуемые ближайшие улучшения:

- единый централизованный обработчик ошибок для slash-команд;
- выделение сервисного слоя (`services/*`) для бизнес-логики;
- покрытие критичных сценариев тестами (особенно private lifecycle);
- улучшение observability (структурированные логи, команда статуса).

## Частые проблемы

- Команды не видны:
  - проверить успешную синхронизацию `bot.tree.sync()` и права приложения.
- Бот не запускается:
  - проверить наличие `SECRET` в `.env`.
- Приватки создаются не там:
  - проверить `rooms_category` в `guilds.json`.
- Команды `/private` не выполняются:
  - убедиться, что они вызваны в `control_text_channel`.
